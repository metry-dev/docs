---
title: "Metry.init() — Initialize and configure the Metry SDK"
sidebarTitle: "init()"
description: "Configure and start the Metry SDK. Call once from client-side app startup before using any other method."
---

Call `Metry.init()` once when your application starts, before any calls to `identify()`, `step()`, or other methods. It creates the internal tracker instance, starts the auto-flush timer, and registers page-lifecycle listeners for automatic beacon sending on tab hide and page unload. If you call `init()` a second time, the previous instance is torn down first, making it safe to use under React Strict Mode and Next.js Fast Refresh.

```typescript
Metry.init(config: InitConfig): void
```

## Parameters

<ParamField body="apiKey" type="string" required>
  Your publishable API key. This value is embedded in every outbound request and is safe to include in browser code.
</ParamField>

<ParamField body="endpoint" type="string" default="https://ingest.metry.dev/v1/events">
  The URL of your ingest endpoint. You usually do not need to set this because the SDK defaults to Metry's hosted ingest endpoint. If you override it, it must use `https://`. Plain `http://` is only permitted when the hostname is `localhost`, `127.0.0.1`, or `[::1]`. An invalid or insecure URL causes `init()` to throw synchronously.
</ParamField>

<ParamField body="flushInterval" type="number" default={5000}>
  Milliseconds between automatic flushes. Increase this value to reduce request frequency on low-traffic pages, or decrease it for near-real-time delivery. The timer does not start on the server.
</ParamField>

<ParamField body="maxBatchSize" type="number" default={50}>
  Maximum number of events to include in a single request. When the queue reaches this size, a flush is triggered immediately regardless of the timer. Values above `100` are capped to `100` with a one-time warning.
</ParamField>

<ParamField body="debug" type="boolean" default={false}>
  When `true`, the SDK emits `[Metry]` log messages to the browser console, including the resolved configuration on init and session resets. Warnings (`[Metry - Warning]`) fire regardless of this setting. See [Debugging](/configuration/debugging).
</ParamField>

## Returns

`void`

## Examples

### Minimal setup

```typescript
import { Metry } from '@metry-dev/sdk'

Metry.init({
  apiKey: 'pk_your_publishable_key',
})
```

### Custom flush tuning

```typescript
Metry.init({
  apiKey: 'pk_your_publishable_key',
  flushInterval: 10_000,
  maxBatchSize: 25,
  debug: process.env.NODE_ENV !== 'production',
})
```

### Next.js App Router

See the [Next.js guide](/guides/nextjs) for the recommended `<MetryProvider>` pattern. A minimal version:

```tsx
'use client'
import { useEffect } from 'react'
import { Metry } from '@metry-dev/sdk'

export function MetryProvider({ children }: { children: React.ReactNode }) {
  useEffect(() => {
    Metry.init({ apiKey: process.env.NEXT_PUBLIC_METRY_KEY! })
    return () => Metry.destroy()
  }, [])
  return <>{children}</>
}
```

<Note>
  `Metry.init()` is browser-only. The package has no import-time side effects, but call `init()` and tracking methods from client-side code. Server runtimes skip the flush timer and page-lifecycle listeners; in runtimes with `fetch`, manual flushes can still attempt network requests.
</Note>

<Warning>
  `apiKey` is required. `endpoint` is optional and defaults to `https://ingest.metry.dev/v1/events`. If you set `endpoint`, it must use `https://` (or `http://` for localhost), otherwise `init()` throws immediately.
</Warning>

<Tip>
  Calling `init()` a second time is safe — the previous instance is destroyed first. This behaviour is intentional and means you do not need to guard against double-initialisation in development environments that use React Strict Mode.
</Tip>
