A funnel in Metry is a sequence of named `step()` events that represent meaningful milestones in a user's journey — for example, signing up, completing a profile, and activating a core feature. Each call to `step()` creates an independent event record with a timestamp, a user ID (if the user has been identified), and any properties you attach. Your analytics tool then orders these events chronologically to visualize where users drop off.

## Anatomy of a funnel

The Metry SDK records funnel steps with `Metry.step(name, properties)`. The `name` is the event identifier that your analytics tool groups by, and `properties` is an optional plain object of JSON-serializable metadata. Segment-level traits — like a user's plan or acquisition source — should be attached via `Metry.identify()` so they appear automatically on every event without repeating them on each `step()` call.

## Example: onboarding funnel

The following funnel tracks a new user from signup through full activation across five stages.

```
signup_completed → profile_filled → first_project_created → invited_teammate → activated
```

### Track each step

Call `Metry.step()` at the point in your code where each milestone is reached. Pass event-specific properties — such as the current plan or acquisition source — directly on the event.

```ts
import { Metry } from '@metry-dev/sdk'

// Step 1 — user completes the signup form
Metry.step('signup_completed', {
  plan: 'trial',
  source: 'product_hunt',
})

// Step 2 — user fills in their profile
Metry.step('profile_filled', {
  plan: 'trial',
  source: 'product_hunt',
})

// Step 3 — user creates their first project
Metry.step('first_project_created', {
  plan: 'trial',
  source: 'product_hunt',
})

// Step 4 — user invites a teammate
Metry.step('invited_teammate', {
  plan: 'trial',
  source: 'product_hunt',
})

// Step 5 — user completes activation
Metry.step('activated', {
  plan: 'trial',
  source: 'product_hunt',
})
```

### Attach traits with identify()

Repeating `plan` and `source` on every `step()` call works, but a cleaner approach is to attach those segment-level properties once via `Metry.identify()`. Traits set there are included on every subsequent event automatically.

```ts
import { Metry } from '@metry-dev/sdk'

// Identify the user with segment traits
Metry.identify('user_0042', {
  email: 'alice@example.com',
  name: 'Alice',
  plan: 'trial',
  source: 'product_hunt',
})

// Steps no longer need to repeat plan or source
Metry.step('signup_completed')
Metry.step('profile_filled')
Metry.step('first_project_created')
Metry.step('invited_teammate')
Metry.step('activated')
```

### Complete working example

The example below mirrors the pattern used in the Metry seed script — identify the user, then walk them through each funnel step with relevant properties.

```ts
import { Metry } from '@metry-dev/sdk'

Metry.init({
  apiKey: process.env.NEXT_PUBLIC_METRY_KEY!,
  endpoint: process.env.NEXT_PUBLIC_METRY_ENDPOINT!,
})

async function trackOnboarding(user: {
  id: string
  email: string
  name: string
  plan: string
  source: string
}) {
  // Associate all events with this user and their segment traits
  Metry.identify(user.id, {
    email: user.email,
    name: user.name,
    plan: user.plan,
    source: user.source,
  })

  // Record each step as the user completes it
  Metry.step('signup_completed', { plan: user.plan, source: user.source })
  Metry.step('profile_filled', { plan: user.plan, source: user.source })
  Metry.step('first_project_created', { plan: user.plan, source: user.source })
  Metry.step('invited_teammate', { plan: user.plan, source: user.source })
  Metry.step('activated', { plan: user.plan, source: user.source })

  // Force-send before the user navigates away (optional)
  await Metry.flush()
}
```

## Naming best practices

Following a consistent naming convention makes funnels easier to query and compare over time.

- **Use `snake_case`** — `signup_completed`, not `SignupCompleted` or `signup-completed`
- **Use past-tense verbs** — events describe something that already happened (`profile_filled`, not `fill_profile`)
- **Order logically** — name steps to reflect the intended sequence so they sort naturally in your analytics tool
- **Be specific** — `first_project_created` is more useful than `project_event` when filtering

## FAQ

<AccordionGroup>
  <Accordion title="What if a user skips a step?">
    Steps are independent events — Metry does not enforce order or require every
    step to be present. If a user never reaches `invited_teammate`, that event
    simply won't exist in your dataset. Your analytics tool will surface this as
    a drop-off between `first_project_created` and `invited_teammate` when you
    build a funnel report.
  </Accordion>

  <Accordion title="Can I track the same step multiple times?">
    Yes. Each call to `Metry.step()` creates a new event record with its own
    timestamp. If a user completes `first_project_created` twice, two events are
    recorded. Most funnel tools let you configure whether to count the first
    occurrence only or all occurrences — check your analytics platform's funnel
    settings to choose the behavior that matches your use case.
  </Accordion>

  <Accordion title="How do I track anonymous users?">
    You can call `Metry.step()` before `Metry.identify()`. In that case the
    events are recorded with a `userId` of `null` and a session ID generated at
    `init()` time. Once the user signs in and you call `Metry.identify()`, all
    subsequent events are attributed to that user. Anonymous pre-identification
    events are not retroactively linked — if you need stitching, handle it in
    your data warehouse using the session ID.
  </Accordion>
</AccordionGroup>
