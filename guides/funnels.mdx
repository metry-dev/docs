A funnel in Metry is an ordered sequence of named steps — for example, `signup_completed → profile_filled → first_project_created → invited_teammate → activated`. You define the funnel in your Metry dashboard, then instrument the same step names in your code with `Metry.step()`. Metry matches events to the funnel by name and shows you conversion rate, drop-off, and time-to-complete for each stage.

## How it works

There are two sides to every funnel:

1. **In the dashboard:** you create a funnel and define its steps in order. Step names are plain strings (1–100 characters) — no schema, no predefined vocabulary.
2. **In your code:** you call `Metry.step(name)` at each point where a user reaches one of those milestones. The name must match a step you defined in the dashboard.

Each call to `step()` records one event with a timestamp, the identified user (if any), and any properties you attach. Metry orders these events chronologically per user and builds the funnel visualization on the server side.

<Note>
  You don't need to call steps in order, and you don't need to call every step. Users who skip steps show up as drop-offs in the funnel between the last step they hit and the next one they didn't.
</Note>

## Traits vs properties

You attach metadata to events in two places, and the distinction matters for how you analyze your funnel later.

- **Traits** go on the user via `Metry.identify()`. Use them for things that are true about the user across all events — plan, acquisition source, email, name. Traits are automatically attached to every subsequent event until you call `identify()` again or `reset()`.
- **Properties** go on a single event via the second argument to `Metry.step()`. Use them for things specific to that one moment — which plan the user selected *at this step*, the button variant they clicked, a referrer from the current page.

As a rule of thumb: if you'd have to pass the same value to every `step()` call, it belongs in `identify()`.

```ts
// Set once — attached to every event that follows
Metry.identify('user_0042', {
  email: 'alice@example.com',
  plan: 'trial',
  source: 'product_hunt',
})

// Per-event properties only — plan and source come from traits
Metry.step('signup_completed')
Metry.step('plan_selected', { selected_plan: 'pro' })
```

## A typical onboarding funnel

Here's what a five-step onboarding funnel looks like when instrumented across a real application. Each call happens in the context where the milestone actually occurs — not all in one block.

```ts
// After signup form submits successfully — in your signup component
Metry.identify(user.id, {
  email: user.email,
  plan: user.plan,
  source: user.source,
})
Metry.step('signup_completed')
```

```ts
// After the user saves their profile — in your profile form handler
Metry.step('profile_filled')
```

```ts
// After the first project is created — in your project creation flow
Metry.step('first_project_created', { template: 'blank' })
```

```ts
// After a teammate invite is sent — in your team invite component
Metry.step('invited_teammate', { invite_method: 'email' })
```

```ts
// Once the user has met your activation criteria — usually triggered server-side
// from your own logic and signaled to the client
Metry.step('activated')
```

Once these events are flowing, create a funnel in your Metry dashboard with these five step names in order. Metry will immediately start showing conversion, drop-off, and timing for every user who enters.

## Naming best practices

Consistent names make funnels easier to read and compare over time.

- **Use `snake_case`** — `signup_completed`, not `SignupCompleted` or `signup-completed`.
- **Use past-tense verbs** — events describe something that already happened. `profile_filled`, not `fill_profile`.
- **Be specific** — `first_project_created` is more useful than `project_event` when you want to segment later.
- **Keep names stable** — renaming a step in your code without updating the funnel in your dashboard will break the match. Treat step names like API contracts.

## FAQ

<AccordionGroup>

<Accordion title="Do I have to create the funnel in the dashboard before I start tracking?">
No. You can start sending `step()` events first and create the funnel afterward — Metry stores all events regardless. But you won't see funnel analytics until you define the funnel and its step order in the dashboard.
</Accordion>

<Accordion title="What if a user skips a step?">
Steps are independent events — Metry doesn't enforce order or require every step to be present. If a user never reaches `invited_teammate`, no event is recorded for it, and Metry's funnel view surfaces the drop-off between the previous step they hit and this one.
</Accordion>

<Accordion title="Can I track the same step multiple times?">
Yes. Each `step()` call creates a new event with its own timestamp. If a user completes `first_project_created` twice, two events are recorded. Metry's funnel view counts each user once per step, using their first occurrence by default.
</Accordion>

<Accordion title="How do I track anonymous users?">
You can call `Metry.step()` before `Metry.identify()`. Events are recorded with `userId: null` and the session ID generated at `init()` time. Once the user signs in and you call `identify()`, subsequent events are attributed to that user — but earlier anonymous events are not retroactively linked to them. If you need to stitch anonymous and identified sessions together, do it downstream using the session ID.
</Accordion>

<Accordion title="Can I use the same step name in multiple funnels?">
Yes. Step names are strings — they have no owning funnel. If `signup_completed` is the first step in your main onboarding funnel and also appears in a separate "trial-to-paid" funnel, a single `Metry.step('signup_completed')` call feeds both.
</Accordion>

</AccordionGroup>
