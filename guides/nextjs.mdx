Metry runs entirely in the browser, so in Next.js you initialize it inside a client component and wrap your app in a provider. This guide covers the App Router setup. If you're on Pages Router, see the note at the bottom.

## Setup

<Steps>

<Step title="Install the SDK">
Add `@metry-dev/sdk` to your project.

<CodeGroup>
```bash npm
npm install @metry-dev/sdk
```
```bash pnpm
pnpm add @metry-dev/sdk
```
```bash yarn
yarn add @metry-dev/sdk
```
</CodeGroup>

Then add your publishable key to `.env.local`:

```bash .env.local
NEXT_PUBLIC_METRY_KEY=pk_...
```

You can find your key under **Settings → API Keys** in your Metry dashboard.
</Step>

<Step title="Create a Metry provider component">
Create `app/providers/metry-provider.tsx`. It initializes the SDK once on mount and tears it down on unmount.

```tsx app/providers/metry-provider.tsx
'use client'

import { useEffect } from 'react'
import { Metry } from '@metry-dev/sdk'

export function MetryProvider({ children }: { children: React.ReactNode }) {
  useEffect(() => {
    Metry.init({
      apiKey: process.env.NEXT_PUBLIC_METRY_KEY!,
    })

    return () => {
      Metry.destroy()
    }
  }, [])

  return <>{children}</>
}
```

<Note>
  `Metry.init()` is idempotent. If an instance is already running — during React Strict Mode double-invocation or Next.js Fast Refresh — the previous instance is torn down automatically before a new one starts. You won't get duplicate flushes or event listeners.
</Note>
</Step>

<Step title="Add the provider to your root layout">
Wrap your app's `children` with `<MetryProvider>` in `app/layout.tsx`.

```tsx app/layout.tsx
import { MetryProvider } from './providers/metry-provider'

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <body>
        <MetryProvider>{children}</MetryProvider>
      </body>
    </html>
  )
}
```

`layout.tsx` can remain a server component — only `MetryProvider` itself needs `'use client'`.
</Step>

<Step title="Track funnel steps">
Call `Metry.step()` from any client component to record an event. The name should match a step in one of your funnels. The optional second argument is a plain object of properties to attach.

```tsx app/components/signup-form.tsx
'use client'

import { Metry } from '@metry-dev/sdk'

export function SignupForm() {
  async function handleSubmit() {
    // ...your signup logic

    Metry.step('signup_completed', {
      plan: 'pro',
      source: 'google_ads',
    })
  }

  return <button onClick={handleSubmit}>Create account</button>
}
```
</Step>

<Step title="Identify users">
Identify a user as soon as you know who they are — usually right after login. The cleanest pattern is a small client component that reads the session and calls `identify()` once:

```tsx app/providers/metry-identity.tsx
'use client'

import { useEffect } from 'react'
import { Metry } from '@metry-dev/sdk'
import { useSession } from '@/lib/auth'

export function MetryIdentity() {
  const { data: session } = useSession()

  useEffect(() => {
    if (!session?.user) return

    Metry.identify(session.user.id, {
      email: session.user.email,
      plan: session.user.plan,
    })
  }, [session?.user?.id])

  return null
}
```

Drop `<MetryIdentity />` inside `<MetryProvider>` in your layout. It runs once the session is available and re-identifies if the user changes.

If your auth flow is fully client-side, you can also call `Metry.identify()` directly from your login handler — whichever fits your stack.
</Step>

<Step title="Track route changes">
Next.js App Router doesn't trigger a full page load between routes. To record route changes as events, hook into `usePathname`:

```tsx app/providers/metry-page-tracker.tsx
'use client'

import { useEffect } from 'react'
import { usePathname } from 'next/navigation'
import { Metry } from '@metry-dev/sdk'

export function MetryPageTracker() {
  const pathname = usePathname()

  useEffect(() => {
    Metry.step('page_view', { path: pathname })
  }, [pathname])

  return null
}
```

Drop `<MetryPageTracker />` inside `<MetryProvider>`. Every navigation will fire a `page_view` event with the new path.

<Tip>
  Only instrument `page_view` if you actually want it as a tracked step. Metry is designed around explicit funnel steps — for most products, tracking key actions like `signup_completed` or `checkout_started` gives you more useful funnels than raw pageview data.
</Tip>
</Step>

<Step title="Handle logout">
Call `Metry.reset()` when the user signs out. This clears the user ID, traits, and pending event queue, and starts a new session.

```tsx app/components/logout-button.tsx
'use client'

import { Metry } from '@metry-dev/sdk'

export function LogoutButton() {
  async function handleLogout() {
    // ...your sign-out logic

    Metry.reset()
  }

  return <button onClick={handleLogout}>Sign out</button>
}
```
</Step>

</Steps>

## Server Components and Server Actions

Metry is browser-only. Avoid importing or calling it from Server Components, Route Handlers, or Server Actions. Imports have no runtime side effects, but server calls are not a supported tracking path: browser lifecycle hooks will not run, and modern server runtimes with `fetch` may still attempt network requests. If you need to record an event as the result of a server action, either:

- Return a signal from the action and call `Metry.step()` in the client component that invoked it, or
- Call `Metry.step()` optimistically before invoking the action.

## Using Pages Router?

The setup is similar — initialize in `pages/_app.tsx` inside a `useEffect`, and use `router.events` from `next/router` for route-change tracking. A dedicated Pages Router guide is coming; for now, the [Quickstart](/quickstart) covers the core flow.
