In a single-page app the page never fully reloads, so you initialize Metry once at startup and keep the instance alive for the whole session. Route changes, logins, and logouts all happen through explicit SDK calls rather than page navigation.

This guide uses React with Vite and React Router as the primary example. The same patterns apply to Create React App or any other React setup — only the environment variable prefix changes (`VITE_` vs `REACT_APP_`).

## Setup

<Steps>

<Step title="Install the SDK">
Add `@metry-dev/sdk` to your project.

<CodeGroup>
```bash npm
npm install @metry-dev/sdk
```
```bash pnpm
pnpm add @metry-dev/sdk
```
```bash yarn
yarn add @metry-dev/sdk
```
</CodeGroup>

Then add your publishable key to `.env`:

```bash .env
VITE_METRY_KEY=pk_...
```

You can find your key under **Settings → API Keys** in your Metry dashboard.
</Step>

<Step title="Initialize at app startup">
Call `Metry.init()` at your entry point, before rendering. This starts the flush loop and registers the page-unload listeners.

```tsx src/main.tsx
import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import { Metry } from '@metry-dev/sdk'
import { App } from './App'

Metry.init({
  apiKey: import.meta.env.VITE_METRY_KEY,
})

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>,
)
```

<Note>
  `Metry.init()` is idempotent. React Strict Mode double-invokes effects in development, and Vite's hot module replacement can re-run your entry file — both are handled automatically, with no duplicate listeners or flushes.
</Note>
</Step>

<Step title="Track funnel steps">
Call `Metry.step()` whenever a user completes a step in one of your funnels. The name should match a step you've defined in your Metry dashboard. The second argument is an optional object of properties.

```tsx src/components/SignupForm.tsx
import { Metry } from '@metry-dev/sdk'

export function SignupForm() {
  async function handleSubmit() {
    // ...your signup logic

    Metry.step('signup_completed', {
      plan: 'pro',
      source: 'google_ads',
    })
  }

  return <button onClick={handleSubmit}>Create account</button>
}
```
</Step>

<Step title="Identify users after login">
Once you know who the user is, call `Metry.identify()` with a stable user ID and any traits you want attached to subsequent events.

```ts
import { Metry } from '@metry-dev/sdk'

async function onLoginSuccess(user: {
  id: string
  email: string
  plan: string
}) {
  Metry.identify(user.id, {
    email: user.email,
    plan: user.plan,
  })
}
```

The identity is held in memory for the current page session. Call `identify()` again to update traits, or `reset()` on logout to clear it. A hard reload clears it automatically — re-identify as soon as your app knows who the user is.
</Step>

<Step title="Track route changes">
SPAs don't fire page loads on navigation, so you need to emit an event yourself when the route changes. With React Router v6:

```tsx src/components/RouteTracker.tsx
import { useEffect } from 'react'
import { useLocation } from 'react-router-dom'
import { Metry } from '@metry-dev/sdk'

export function RouteTracker() {
  const location = useLocation()

  useEffect(() => {
    Metry.step('page_view', { path: location.pathname })
  }, [location.pathname])

  return null
}
```

Render `<RouteTracker />` once inside your router:

```tsx src/App.tsx
import { BrowserRouter, Routes } from 'react-router-dom'
import { RouteTracker } from './components/RouteTracker'

export function App() {
  return (
    <BrowserRouter>
      <RouteTracker />
      <Routes>
        {/* your routes */}
      </Routes>
    </BrowserRouter>
  )
}
```

<Tip>
  Only instrument `page_view` if you want it as a tracked step. Metry is designed around explicit funnel steps — tracking the actions that matter (`signup_completed`, `plan_selected`, `checkout_started`) usually produces more useful funnels than raw pageviews.
</Tip>
</Step>

<Step title="Handle logout">
Call `Metry.reset()` when the user signs out. This clears the user ID, traits, and any pending events, and starts a new session.

```ts
import { Metry } from '@metry-dev/sdk'

async function onLogout() {
  // ...your sign-out logic

  Metry.reset()
}
```
</Step>

</Steps>

## Advanced: teardown in micro-frontends

Most SPAs don't need to call `Metry.destroy()` — the SDK lives for the lifetime of the page, and the automatic flush on `pagehide` handles the final send.

The exception is a micro-frontend shell that loads and unloads sub-applications at runtime. In that case, call `Metry.destroy()` in your cleanup hook to stop the flush loop, remove listeners, and attempt one final flush:

```tsx
import { Metry } from '@metry-dev/sdk'

Metry.init({ apiKey: import.meta.env.VITE_METRY_KEY })

const root = createRoot(document.getElementById('root')!)
root.render(<App />)

export function unmount() {
  Metry.destroy()
  root.unmount()
}
```

After `destroy()`, call `Metry.init()` again before using any other method.
