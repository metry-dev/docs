The Metry SDK exposes two levels of console output to help you diagnose problems: opt-in debug logs that trace normal SDK activity, and always-on warnings that fire whenever something unexpected happens. Most integration issues can be identified by reading these messages before consulting this page.

## Enabling debug mode

Pass `debug: true` to `Metry.init()` to turn on verbose logging:

```js
Metry.init({
  apiKey: process.env.NEXT_PUBLIC_METRY_KEY,
  endpoint: 'https://ingest.example.com/v1/events',
  debug: true,
})
```

When debug mode is active, the SDK writes messages prefixed with `[Metry]` to the browser console. These messages cover:

- Initialization parameters (`endpoint`, `flushInterval`, `maxBatchSize`)
- Flush triggers (timer-based and threshold-based)
- Batch sizes at the time of each send
- Successful HTTP responses
- Failed HTTP responses and re-queue decisions

<Tip>
  Disable debug mode in production builds using an environment check:

  ```js
  Metry.init({
    apiKey: process.env.NEXT_PUBLIC_METRY_KEY,
    endpoint: 'https://ingest.example.com/v1/events',
    debug: process.env.NODE_ENV !== 'production',
  })
  ```
</Tip>

## SDK warnings

Some messages are always written to the console, regardless of the `debug` setting. These are prefixed with `[Metry - Warning]` and indicate a condition that requires your attention:

| Condition | Warning message |
|-----------|----------------|
| Event name is empty or over 100 characters | `track() requires an event name between 1 and 100 characters` |
| `properties` argument is not a plain object | `track() properties must be a plain object` |
| `navigator.sendBeacon` was rejected by the browser | `navigator.sendBeacon rejected by browser` |
| Endpoint returned a 4xx response | `Endpoint rejected batch with status <N>; dropping events` |
| Endpoint returned a 5xx response | `Endpoint returned status <N>; re-queuing batch` |
| Payload could not be JSON-serialized | `Failed to serialize batch (circular reference or unserializable value); dropping events` |
| `maxBatchSize` exceeds 100 | `maxBatchSize exceeds 100; capping payload batches to 100` |

## Common issues

<AccordionGroup>
  <Accordion title="Events are not being sent">
    Check that `Metry.init()` was called before any other SDK method. Open the browser console and look for `[Metry - Warning]` messages â€” they usually identify the root cause. Also verify that your `endpoint` URL is reachable from the browser by opening it directly or using the Network tab in DevTools.
  </Accordion>

  <Accordion title="I'm getting 'Call .init() first'">
    You called a Metry method â€” such as `Metry.step()` or `Metry.identify()` â€” before `Metry.init()` completed. Initialize the SDK once at application startup, before any components or event handlers that use the SDK run.
  </Accordion>

  <Accordion title="Events are dropped on page unload">
    On modern browsers, the SDK automatically uses `navigator.sendBeacon` when the page becomes hidden (via the `visibilitychange` and `pagehide` events). If you have a custom unload handler that conflicts with the SDK lifecycle, call `Metry.sendBeacon()` manually from your handler to drain the queue before the page closes.
  </Accordion>

  <Accordion title="My endpoint is rejecting requests (4xx)">
    The SDK drops the batch on any 4xx response and does not retry. Check that your `apiKey` is correct and that the `endpoint` URL matches what your ingest service expects. Enable debug mode to log the full request payload and inspect exactly what is being sent.
  </Accordion>

  <Accordion title="maxBatchSize above 100 is silently capped">
    The SDK enforces a hard cap of 100 events per HTTP request. If you set `maxBatchSize` to a value greater than 100, a `[Metry - Warning]` message is logged once and each outgoing batch is capped at 100 events. Your `maxBatchSize` value is not modified â€” only the effective batch size per request is limited.
  </Accordion>

  <Accordion title="I see duplicate events in Strict Mode">
    `Metry.init()` safely tears down the previous instance when called again, so double-invocation under React Strict Mode does not cause duplicate sends. If you are still seeing duplicate events, check that `Metry.identify()` and `Metry.step()` are called only once per user action â€” for example, ensure they are not placed inside a `useEffect` that runs twice due to Strict Mode without a corresponding cleanup.
  </Accordion>
</AccordionGroup>
