The Metry SDK exposes two kinds of console output to help you diagnose problems:

- **Debug logs** (`[Metry] ...`) — opt-in via `debug: true`, used for tracing normal SDK activity.
- **Warnings** (`[Metry - Warning] ...`) — always on, fired whenever the SDK encounters something unexpected.

Most integration issues can be identified by reading these messages before consulting this page.

## Enable debug mode

Pass `debug: true` to `Metry.init()`:

```ts
Metry.init({
  apiKey: process.env.NEXT_PUBLIC_METRY_KEY!,
  debug: true,
})
```

When debug mode is active, the SDK writes messages prefixed with `[Metry]` to the browser console, including:

- The resolved configuration on init (endpoint, flush interval, batch size)
- Session resets
- Notices about unsupported browser APIs (e.g. missing `navigator.sendBeacon`)

<Tip>
  Disable debug mode in production. A common pattern:

```ts
  Metry.init({
    apiKey: process.env.NEXT_PUBLIC_METRY_KEY!,
    debug: import.meta.env.DEV, // or: process.env.NODE_ENV !== 'production'
  })
```
</Tip>

## Warning messages

These messages are always written to the console, regardless of the `debug` setting. Each one corresponds to a specific condition in the SDK.

| Condition | Warning |
|-----------|---------|
| Event name is empty, not a string, or over 100 characters | `track() requires an event name between 1 and 100 characters` |
| `properties` argument is not a plain object | `track() properties must be a plain object` |
| `maxBatchSize` config exceeds 100 | `maxBatchSize exceeds 100; capping payload batches to 100` |
| Payload could not be JSON-serialized (circular ref, etc.) | `Failed to serialize batch (circular reference or unserializable value); dropping events` |
| Session ID was invalid and had to be regenerated | `sessionId is invalid; regenerating before send` |
| Endpoint returned a 4xx response | `Endpoint rejected batch with status <N>; dropping events` |
| Endpoint returned a 5xx response | `Endpoint returned status <N>; re-queuing batch` |
| Network error while sending (fetch rejected) | `Failed to send payload, re-queuing batch` |
| `navigator.sendBeacon` was rejected by the browser | `navigator.sendBeacon rejected by browser` |

Initialization errors throw rather than warn. The most common ones:

- `Missing required configuration` — `apiKey` (or `endpoint`, if you're self-hosting) was not provided.
- `Invalid endpoint URL` — the `endpoint` string couldn't be parsed as a URL.
- `Endpoint must use https:// (http:// allowed for localhost)` — only HTTPS endpoints are accepted in production; `http://` is permitted only for `localhost` or `127.0.0.1` during local development.

## Verifying your integration

After you call `Metry.step()` for the first time, you can confirm events are flowing in three ways:

1. **Network tab.** Open DevTools → Network and filter for `events`. You should see a `POST` request to the ingest endpoint within 5 seconds of the call (sooner if your batch fills). A `204` or `200` response means the event was accepted.
2. **Metry dashboard.** Head to the **Recent users** panel on your project overview. Your user should appear within a minute of the event firing.
3. **Debug mode.** Turn on `debug: true` to see the SDK's init log and any warnings in the console.

If none of these show activity, start at the top of the next section.

## Common issues

<AccordionGroup>

<Accordion title="Events are not being sent">
Most common causes, in order:

1. `Metry.init()` wasn't called, or was called after `Metry.step()`. Initialize the SDK at app startup, before any handlers that track events.
2. The `apiKey` is wrong or disabled. Check **Settings → API Keys** in your dashboard. Keys can be deactivated — make sure the one you're using is marked active.
3. A 4xx response is dropping your batches silently-ish (there's a warning, but it's easy to miss). Look for `[Metry - Warning] Endpoint rejected batch with status 401`.
4. Your browser extension (ad-blocker, privacy tool) is blocking the request. Check the Network tab — blocked requests show as `(blocked)`.
</Accordion>

<Accordion title="'Call .init() first' error">
A Metry method — `step()`, `identify()`, `flush()`, `reset()`, or `destroy()` — was called before `Metry.init()`. This usually happens in one of two situations:

- A component imports and calls `Metry.step()` at module top level before your app's entry file runs `init()`.
- In Next.js, a Server Component or Route Handler tried to use the SDK. Metry is browser-only — move the call into a Client Component.
</Accordion>

<Accordion title="Events sent near page unload aren't arriving">
The SDK hooks `visibilitychange` and `pagehide` automatically and uses `navigator.sendBeacon` to drain the queue. If events around unload still go missing, check for:

- `[Metry - Warning] navigator.sendBeacon rejected by browser` — typically means the payload is too large (most browsers cap beacons at 64KB). Reduce `maxBatchSize` so each batch fits under the cap.
- A custom unload handler in your app that preempts the SDK. If you control the unload path explicitly, call `Metry.sendBeacon()` from your handler to drain the queue.
- Mobile Safari, which is notoriously aggressive about killing background tabs. Some event loss on mobile is unavoidable.
</Accordion>

<Accordion title="Endpoint is rejecting requests with 4xx">
The SDK drops the batch on any 4xx response — it doesn't retry, since retrying a client error would just fail again.

Check:
- `401 Unauthorized` — the `apiKey` is wrong, disabled, or doesn't belong to the project your endpoint is expecting.
- `429 Too Many Requests` — you've exceeded your plan's rate limit. Check your usage in the dashboard.
- `400 Bad Request` — the payload shape doesn't match what the ingest endpoint expects. This usually only happens if you're self-hosting an older ingest version.
</Accordion>

<Accordion title="`maxBatchSize` above 100 is being capped">
The SDK enforces a hard ceiling of 100 events per HTTP request. If you set `maxBatchSize` higher than 100, the warning fires once and each outgoing batch is capped at 100. Your configured value isn't mutated — only the effective batch size per request is limited.

This cap exists because `navigator.sendBeacon` has a browser-enforced payload size limit, and larger batches cause beacon delivery to fail silently on page unload.
</Accordion>

<Accordion title="Duplicate events under React Strict Mode">
`Metry.init()` is idempotent — calling it twice tears down the previous instance, so Strict Mode double-invoke won't cause duplicate sends from init itself.

Duplicate `step()` events under Strict Mode usually come from calling `step()` inside a `useEffect` without a guard. In development, Strict Mode intentionally runs effects twice to help catch non-idempotent logic. Options:

- Move the call into an event handler instead of a `useEffect` where possible.
- Use a ref to guard against the second run:

```tsx
  const tracked = useRef(false)
  useEffect(() => {
    if (tracked.current) return
    tracked.current = true
    Metry.step('page_view', { path: location.pathname })
  }, [location.pathname])
```

Strict Mode only runs in development, so this won't affect production users either way — but it's worth handling cleanly so your dev logs stay accurate.
</Accordion>

<Accordion title="Events are attributed to the wrong user">
If you see events for `user_A` showing up under `user_B` (or under no user), it's almost always one of these:

- You called `Metry.identify()` with the new user but didn't call `Metry.reset()` first. `identify()` overwrites traits but doesn't start a new session — events recorded between logins can share a session ID.
- Traits from a previous login are carrying over because you didn't call `reset()` at logout.

Call `Metry.reset()` on every logout. Call `Metry.identify()` on every login.
</Accordion>

</AccordionGroup>
